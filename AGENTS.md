# Best Practices & AI Agent Instructions

This repository follows a strict architectural pattern to maintain modularity, scalability, and clean boundaries. AI coding agents **MUST** read and adhere to these guidelines when suggesting or implementing changes.

## 1. Feature-Sliced Design (FSD)

We use a variation of the Feature-Sliced Design (FSD) approach.

*   **Rule:** Domain-specific UI logic and components MUST reside in the `features/<feature-name>/components/` directory, NOT in the root `components/` or `app/` directories.
*   **Example:**
    *   Flashcard components go in `features/flashcards/components/`
    *   Discussion components go in `features/community/components/`
    *   Mock exam wrappers go in `features/mock-exams/components/`
    *   Authentication views go in `features/auth/components/`
*   **Rule:** The root `components/` directory should ONLY contain generic, reusable UI elements (e.g., buttons, inputs, layout wrappers).
*   **Rule:** The `app/` directory should ONLY contain Next.js App Router files (e.g., `page.tsx`, `layout.tsx`, `loading.tsx`, `route.ts`). Complex UI implementations should be imported from `features/` or `components/`.

## 2. Centralized Types

To avoid type scattering and circular dependencies, all domain and database types are centralized.

*   **Rule:** Database types (generated by Supabase) MUST be placed in `types/database.types.ts`.
*   **Rule:** Custom application types, interfaces, and mapped types MUST be placed in `types/features.types.ts`.
*   **Rule:** NEVER create local `types.ts` files inside component or feature folders (e.g., do not use `components/discussion/types.ts`). ALWAYS import from `@/types/database.types` or `@/types/features.types`.

## 3. Centralized Data Access (Repository Pattern)

Direct database queries (e.g., calling Supabase client methods like `.from('table').select()`) should be abstracted away from UI components.

*   **Rule:** All data fetching and mutation logic MUST be extracted into `actions.ts` files within their respective feature folders (e.g., `features/flashcards/actions.ts`, `features/community/actions.ts`).
*   **Rule:** UI components (both Client and Server components) should call these exported action functions rather than interacting with the database client directly.
*   **Rule:** Server Actions should be clearly marked with `"use server"` if they are intended to be called directly from Client Components as server actions.

## 4. Single Source of Truth for Database Schema

*   **Rule:** All database migration files (`.sql`) MUST be stored in the `supabase/migrations/` directory.

---
**Verification for AI Agents:**
Before completing any task, ensure you have:
1. Placed domain components in `features/`.
2. Kept `app/` files minimal (mostly routing).
3. Used centralized types from the `types/` folder.
4. Abstracted database queries into feature-level `actions.ts` files.
